# Switch Auto Draw PRD

版本：v0.1
状态：Draft
更新时间：2026-04-25

## 1. 产品概述

`Switch Auto Draw` 是一个基于 `Mac + ESP32-WROOM-32 + Nintendo Switch` 的自动绘图工具。

用户在 Mac 上导入图片，工具会将图片转换为像素矩阵和绘图动作序列，再通过串口下发到 ESP32。ESP32 最终通过 `Bluetooth Classic` 模拟 Switch 控制器，在目标游戏或画板界面中自动完成绘图。

当前产品目标不是做“最快”的自动绘图器，而是优先做一个：

- 可复现
- 可调试
- 可逐步校准
- 稳定优先于速度和效果

## 2. 背景与问题

当前手工在 Switch 画板中复刻图片存在几个问题：

- 重复劳动多，耗时长
- 像素图对齐和颜色切换容易出错
- 人工移动光标稳定性差
- 想复现同一张图时操作成本高

本项目要解决的是：让用户在电脑上完成图片准备和参数配置，再由硬件稳定执行绘图动作。

## 3. 产品目标

### 3.1 MVP 目标

做出一条可验证的最小闭环：

`导入图片 -> 生成像素命令 -> 串口发送 -> ESP32 解析 -> 控制器动作执行`

### 3.2 成功标准

MVP 成功至少满足以下条件：

- 用户可在 Mac 上导入图片并生成预览
- 用户可导出绘图命令文本
- 用户可通过串口把命令逐条发送给 ESP32
- ESP32 能稳定返回 `OK`，不丢命令
- 在接入真实蓝牙手柄层后，Switch 端能完成基础单色绘图

## 5. 目标用户

### 5.1 核心用户

- 熟悉前端或脚本开发的创客用户
- 想在 Switch 画板/创作类游戏里快速复刻图片的用户
- 能接受自己烧录开发板和进行少量校准的早期使用者

### 5.2 非目标用户

- 完全不愿接触开发板和串口工具的普通消费者
- 期待零配置、开箱即用的用户
- 需要跨平台 Windows / Linux 原生 GUI 的用户

## 6. 使用场景

### 6.1 场景 A：本地命令生成

用户在 Mac 选择一张图片，设置目标尺寸和颜色模式，生成预览图和命令文件，确认效果是否可接受。

### 6.2 场景 B：串口实时绘制

用户连接 ESP32，选择串口端口，开始逐条发送绘图命令，让 ESP32 实时执行。

### 6.3 场景 C：后续离线绘制

用户先把图像数据或绘图任务写入 ESP32，之后不依赖 Mac 也能独立执行绘图。

注：场景 C 不是当前仓库已实现功能，是后续版本目标。

## 7. 产品范围

### 7.1 当前版本范围

- 平台：`macOS`
- 桌面端形态：`CLI + Web 原型`
- 硬件主线：`ESP32-WROOM-32 / ESP-32S`
- 通信方式：`Mac USB serial -> ESP32`
- 控制器路线：`ESP32 Bluetooth Classic -> Switch`
- 图像输入：`PNG / JPG / SVG`
- 绘图模式：`黑白 / 固定调色板`

### 7.2 当前明确不做

- Windows 和 Linux GUI 支持
- 自动识别 Switch 游戏画面
- 自动视觉校准
- 任意分辨率高保真重绘
- 复杂路径优化算法
- 云端同步或账号系统

## 8. 用户流程

### 8.1 MVP 标准流程

1. 用户在 Mac 上准备图片
2. 用户运行 CLI 生成预览和命令文件
3. 用户连接 ESP32 并烧录固件
4. 用户运行 smoke test，确认串口 ACK 正常
5. 用户发送真实绘图命令
6. ESP32 接收并执行控制器动作
7. Switch 完成绘图

### 8.2 到货当天建议流程

1. 识别串口设备
2. 烧录固件
3. 查看启动日志
4. 运行 smoke test
5. 再运行 demo 图像命令

### 8.3 目标绘画页初始状态

当前主线先按单一目标绘画场景建模，假设如下：

- 用户已经手动进入绘画页面
- 进入页面时光标停在主画布中心
- 当前默认工具为 `pen`
- 颜色栏固定在右侧
- 工具栏固定在顶部

在这个假设下，正式绘制前需要先完成一次固定校准：

- 将光标从 `center` 移动到 `top-left`
- 之后再以左上角为绘图原点执行扫描线路径

### 8.4 场景 profile 字段

当前需要在 profile 中记录以下场景配置：

- `startCursor`
- `startTool`
- `startColorIndex`
- `centerToTopLeftDx`
- `centerToTopLeftDy`

这些字段的作用是固定当前目标界面，并为后续实机校准保留结构。

## 9. 功能需求

### 9.1 Mac 端

#### 必须支持

- 导入图片
- 指定尺寸
- 指定黑白阈值或调色板
- 生成像素预览
- 生成命令文件
- 枚举串口设备
- 按 ACK 模式发送命令
- 支持暂停、继续、停止
- 能读取并保留目标绘画页的场景 profile

#### 可选支持

- 估算剩余时间
- 记录发送进度
- 导出中间调试数据

### 9.2 ESP32 固件

#### 必须支持

- 串口接收文本命令
- 逐条解析命令
- 正确返回 `OK` 或 `ERR`
- 提供基本动作抽象：移动、按键、回原点、等待

#### 下一阶段必须支持

- Bluetooth Classic 控制器传输层
- Switch 配对与保持连接
- 基础按键与方向输入
- 支持从页面中心校准到左上角的起笔定位

### 9.3 后续 GUI

当前 Web 原型已经支持图片导入、预览、脚本生成与串口发送；未来 Electron 版还需要支持：

- 图片导入
- 参数设置
- 预览
- 串口选择
- 开始 / 暂停 / 停止
- 进度显示

## 10. 协议需求

当前协议采用简单文本命令，便于调试和串口观察。

基础命令集：

- `I`：读取设备信息
- `H`：回原点
- `M dx dy`：相对移动
- `P`：绘制当前格
- `A / B / X / Y`：按钮动作
- `C n`：切换颜色
- `W ms`：等待
- `S`：暂停
- `R`：继续
- `E`：结束

响应要求：

- 每执行完一条命令返回 `OK`
- 若命令非法返回 `ERR <reason>`

## 11. 非功能需求

### 11.1 稳定性

- 串口命令不应批量淹没 ESP32 缓冲区
- 需要优先保证命令不丢失
- 设备异常时必须返回可观察错误

### 11.2 可调试性

- 所有关键步骤都要能在串口日志中观察
- 命令文件必须可人工阅读
- 支持 smoke test，不依赖图像链路也能单独验证固件

### 11.3 可扩展性

- 图像处理、路径生成、协议、串口发送需模块化
- 硬件控制层与协议层分离
- 未来可以替换为 GUI 或离线模式

## 12. 硬件与技术约束

### 12.1 已选定硬件主线

- 开发板：`ESP32-WROOM-32 / ESP-32S`
- 串口桥：`CP2102`
- 操作系统：`macOS`

### 12.2 关键约束

- 当前最终目标依赖 `Bluetooth Classic`
- 当前项目主线不是 `USB HID` 路线
- Switch 输入稳定性比算法效果更重要

## 14. 风险与应对

### 14.1 蓝牙协议风险

风险：

- Switch 对控制器模拟兼容性要求高

应对：

- 尽早验证最小按键输入
- 先做连接和单按键，再做连续绘图

### 14.2 光标漂移风险

风险：

- 单格移动耗时误差会累计，导致画歪

应对：

- 优先支持小尺寸图
- 暴露 `cellMoveDuration` 和 `inputDelay` 配置
- 先固定“中心入场 -> 左上角起笔”的单一场景
- 先稳定再提速

### 14.3 颜色切换风险

风险：

- 不同游戏颜色 UI 差异大

应对：

- 第一版只针对单一目标场景
- 先黑白，后多色
